
CÂU TRẢ LỜI KHẢO SÁT – MỨC 2 & MỨC 3
(Dựa trên cấu trúc source code hiện tại)

### Mức 2 – Lấy dữ liệu DB
1. Dữ liệu nào cần inject vào prompt?
👉 Cần tất cả (Bác sĩ, Dịch vụ, Lịch trống) nhưng phải phụ thuộc vào câu hỏi. 
- Để tư vấn khám: Cần danh sách Dịch vụ + Chuyên khoa + Bác sĩ.
- Để đặt lịch: Cần chính xác thời gian rảnh (TimeSlot) của bác sĩ đó trong tương lai gần.

2. Inject theo cách nào?
👉 Chọn B Chỉ inject khi intent liên quan (tiết kiệm token và tăng tốc độ).
- Lịch trống thay đổi liên tục, nếu "nhồi" tất cả vào mọi prompt sẽ gây quá tải token và phản hồi chậm.
- Trong ChatbotService, khi lấy Intent: 
  + Nếu Intent == "GetDoctorInfo", gọi _unitOfWork.Doctors.GetAllAsync() để nhét list bác sĩ vào.
  + Nếu Intent == "BookAppointment", gọi _unitOfWork.TimeSlots.GetAvailableSlotsAsync(...) ghép vào prompt.

### Mức 3 – Tự đặt lịch
3. Flow đặt lịch qua chat muốn như thế nào?
👉 Kết hợp linh hoạt cả A và B.
- Dặn Gemini trong prompt: "Để đặt lịch, bắt buộc phải có đủ 3 thông tin: Bác sĩ/Chuyên khoa, Ngày khám, Giờ khám".
- Nếu user đưa đủ thông tin 1 câu (Flow B): Gemini trả về JSON chứa đầy đủ Entities (DoctorId, SlotId) để map xuống DB.
- Nếu thiếu thông tin (Flow A): Gemini tự động hỏi lại để gom đủ Entities.

4. Nếu thiếu thông tin mà chưa chọn bác sĩ cụ thể → xử lý thế nào?
👉 Chọn B) Hỏi lại user chọn bác sĩ nào (kèm gợi ý).
- Trong y tế, việc tự ý xếp lịch có thể gây trải nghiệm kém. Bot nên phản hồi: "Hiện tại chuyên khoa Tim mạch chiều thứ 6 đang có Bác sĩ A và Bác sĩ B trống lịch. Bạn muốn ưu tiên khám bác sĩ nào không, hay để hệ thống sắp xếp bác sĩ sớm nhất giúp bạn?".

5. Sau khi đặt lịch thành công, chatbot có cần gửi Notification realtime (SignalR) không?
👉 CÓ, CHẮC CHẮN CẦN.
- Để code gọn và tái sử dụng logic, ChatbotService không nên tự gọi _unitOfWork.Appointments.Add(). Thay vào đó, hãy inject thẳng IAppointmentService vào ChatbotService.
- Khi đặt lịch, chatbot chỉ cần gọi: await _appointmentService.BookAsync(dto, cancellationToken).
- Hàm BookAsync này đã có sẵn logic lưu DB, xử lý transaction và gọi NotificationService.SendNotificationAsync() để bắn SignalR cho cả Admin và Bác sĩ.

PHẦN CƠ SỞ LÝ THUYẾT (Công nghệ cốt lõi sử dụng trong đồ án)

Đồ án được xây dựng theo kiến trúc Web API kết hợp Single Page Application (SPA), kế thừa tư tưởng của mô hình MVC truyền thống bằng cách giữ lại Model và Controller ở Backend, còn tầng View được tách rời hoàn toàn cho Frontend.

Dưới đây là 6 công nghệ quan trọng nhất cần đưa vào báo cáo:

1. React kết hợp TypeScript: Thư viện cốt lõi kết hợp với ngôn ngữ định kiểu tĩnh (static typing) để xây dựng giao diện người dùng (UI) an toàn, dễ bảo trì và tương tác cao (đóng vai trò là tầng View).
2. ASP.NET Core (.NET 9): Framework mạnh mẽ dùng để xây dựng Web API xử lý luồng nghiệp vụ (đóng vai trò là tầng Controller).
3. Entity Framework Core (EF Core): Công cụ ORM giúp thao tác và ánh xạ dữ liệu thành các đối tượng C# (đóng vai trò là tầng Model).
4. SQL Server: Hệ quản trị cơ sở dữ liệu quan hệ, đảm nhiệm việc lưu trữ dữ liệu bền vững và bảo mật cho toàn bộ hệ thống.
5. SignalR: Công nghệ truyền tải dữ liệu thời gian thực (Realtime) để làm tính năng thông báo trực tiếp.
6. Google Gemini API: Mô hình trí tuệ nhân tạo (LLM) tích hợp vào hệ thống để phân tích ngôn ngữ tự nhiên (NLP) cho Chatbot hỗ trợ y tế.

ĐỀ XUẤT CÁC BIỂU ĐỒ TUẦN TỰ (SEQUENCE DIAGRAM) CẦN BỔ SUNG THÊM

Sau khi xem qua danh sách biểu đồ tuần tự bạn đã có (Đăng nhập, Đặt lịch, Chatbot, Đánh giá, Lịch làm việc, Thông tin cá nhân), mình thấy bạn đã làm rất xuất sắc và bao phủ được hầu hết các luồng chính! 

Tuy nhiên, để đồ án thật sự hoàn hảo và bao phủ hết toàn bộ các Module đã xây dựng trong Source Code, bạn có thể bổ sung thêm 3-4 biểu đồ tuần tự sau (ưu tiên theo thứ tự):

1. Biểu đồ tuần tự: Cập nhật kết quả khám và Tạo hồ sơ bệnh án (Nghiệp vụ Y tế cốt lõi)
- Tác nhân: Bác sĩ, Frontend, Backend, Database.
- Luồng chạy: Bác sĩ chọn bệnh nhân đang chờ khám -> Điền thông tin chẩn đoán, toa thuốc -> Frontend gọi API lưu hồ sơ -> Backend cập nhật trạng thái lịch hẹn thành "Đã khám xong" và lưu Hồ sơ bệnh án -> Database lưu trữ -> Trả về thông báo thành công.
- Lý do: Bạn đã có "Đặt lịch khám" nhưng lại chưa có quy trình "Khám bệnh thực tế". Đây là bước đóng vòng đời của một lịch hẹn.

2. Biểu đồ tuần tự: Quy trình Check-in và luân chuyển bệnh nhân (Áp dụng Realtime SignalR)
- Tác nhân: Lễ tân, Bác sĩ, Frontend, Backend, SignalR Hub, Database.
- Luồng chạy: Bệnh nhân đến quầy -> Lễ tân bấm "Check-in" -> Backend lưu trạng thái "CheckedIn" vào DB -> Backend kích hoạt SignalR Hub -> SignalR đẩy thông báo realtime lên màn hình của Bác sĩ phụ trách -> Bác sĩ biết bệnh nhân đã đến và gọi vào phòng.
- Lý do: Thể hiện điểm sáng về mặt công nghệ (WebSockets/SignalR) giải quyết bài toán thời gian thực tại phòng khám.

3. Biểu đồ tuần tự: Soạn thảo và Kiểm duyệt bài viết Y khoa
- Tác nhân: Bác sĩ, Admin, Frontend, Backend, Database.
- Luồng chạy: Bác sĩ viết bài -> Gọi API Lưu nháp/Gửi duyệt -> Trạng thái bài viết là "Pending" -> Admin vào danh sách chờ duyệt -> Admin đọc và bấm "Phê duyệt" -> Backend chuyển trạng thái thành "Published" -> Bài viết hiện lên trang chủ cho Bệnh nhân đọc.
- Lý do: Bao phủ được Module Bài viết (rất nhiều code xử lý) và thể hiện luồng nghiệp vụ kiểm duyệt nhiều vai trò (Role-based Workflow).

4. Biểu đồ tuần tự: Quên mật khẩu và Khôi phục tài khoản (Tích hợp Email Service)
- Tác nhân: Người dùng, Frontend, Backend, Database, SMTP Email Server.
- Luồng chạy: Người dùng nhập Email -> Backend kiểm tra tồn tại -> Backend tạo mã Token bảo mật và lưu DB -> Backend gọi Background Job kết nối SMTP Server -> Gửi Email chứa link reset -> Người dùng click link -> Nhập mật khẩu mới -> API đổi mật khẩu -> Thành công.
- Lý do: Thể hiện được khả năng tích hợp với hệ thống bên thứ 3 (Email Server) và xử lý Background Task.

# CHƯƠNG 5: KIỂM THỬ HỆ THỐNG

## 5.1 Mục tiêu kiểm thử
Quá trình kiểm thử hệ thống được thực hiện nhằm mục đích:
- Đảm bảo các chức năng cốt lõi của hệ thống hoạt động đúng theo yêu cầu phân tích thiết kế.
- Phát hiện và khắc phục các lỗi (bug) phát sinh trong quá trình vận hành luồng nghiệp vụ.
- Đánh giá tính ổn định, bảo mật và khả năng đáp ứng của hệ thống khi tương tác với dữ liệu thực tế.
- Đảm bảo trải nghiệm người dùng (UX/UI) nhất quán và thuận tiện trên các vai trò: Bệnh nhân, Bác sĩ, và Quản trị viên (Admin/Lễ tân).

## 5.2 Môi trường kiểm thử
Quá trình kiểm thử được thực hiện trên môi trường có các thông số kỹ thuật và nền tảng công nghệ như sau:

**Phần cứng (Hardware):**
- CPU: Máy tính cá nhân chip xử lý Intel Core i5 / AMD Ryzen 5 trở lên.
- Bộ nhớ (RAM): 8GB trở lên.

**Phần mềm và Công nghệ (Software & Technologies):**
- **Hệ điều hành:** Windows 10 / Windows 11.
- **Trình duyệt Web:** Google Chrome (v110+), Microsoft Edge.
- **Frontend:** Ứng dụng web tĩnh Single Page Application (SPA) xây dựng bằng thư viện ReactJS, công cụ build Vite, ngôn ngữ TypeScript và CSS Tailwind.
- **Backend:** API Server xây dựng theo kiến trúc RESTful bằng framework ASP.NET Core 8 Web API, ngôn ngữ C#. Hệ thống gửi nhận thông báo thời gian thực bằng SignalR.
- **Cơ sở dữ liệu:** Hệ quản trị CSDL Microsoft SQL Server 2022, thao tác dữ liệu qua Entity Framework Core.
- **Công cụ kiểm thử:** Kiểm thử thủ công (Manual Testing) bằng hộp đen (Black-box) trực tiếp trên giao diện trình duyệt Web và công cụ Postman để kiểm thử luồng dữ liệu API độc lập.

## 5.4 Kết quả kiểm thử
Sau khi tiến hành thực thi các ca kiểm thử (Test Cases) trên môi trường thực tế của hệ thống, thu được kết quả tổng kết như sau:
- Tổng số ca kiểm thử đã thực hiện: **11**
- Số lượng Test Case đạt (Pass): **11** (Đạt tỷ lệ 100%)
- Số lượng Test Case không đạt (Fail): **0**

**Nhận xét quá trình kiểm thử:**
- Các luồng nghiệp vụ cốt lõi như luân chuyển trạng thái lịch khám (Pending -> Confirmed -> CheckedIn -> Completed), quy trình ghi nhận hồ sơ bệnh án và chức năng tích hợp Trí tuệ nhân tạo (Gemini AI) đều hoạt động chính xác theo thiết kế. 
- Hệ thống bắt lỗi tốt (Validation) đối với các trường hợp thao tác sai logic nghiệp vụ (ví dụ: đặt trùng lịch, điền thiếu form khám bệnh), trả về mã lỗi HTTP chuẩn mực kết hợp với thông báo giao diện dễ hiểu mà không gây treo (crash) ứng dụng. 
- Điểm sáng là sự đồng bộ hóa thời gian thực qua SignalR hoạt động rất nhạy và trơn tru trong luồng tiếp đón bệnh nhân.

## 5.5 Kết luận
Qua quá trình kiểm thử hộp đen với các kịch bản sát thực tế, có thể kết luận rằng "Hệ thống đặt lịch khám và đặt dịch vụ y tế trực tuyến tích hợp AI Gemini" đã đáp ứng đầy đủ các yêu cầu về mặt chức năng và kỹ thuật đặt ra ở chương phân tích hệ thống.

Phần mềm vận hành ổn định, bảo đảm tính toàn vẹn dữ liệu, giao diện tương tác tốc độ cao và đảm bảo tốt quy trình làm việc khép kín của một phòng khám từ khâu đặt lịch, tiếp đón cho đến lưu trữ hồ sơ y tế. Sản phẩm hoàn toàn đáp ứng tốt các tiêu chí để có thể đưa vào thực nghiệm và bảo vệ đồ án tốt nghiệp.
